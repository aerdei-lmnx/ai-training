What is an LLM?

Large Language Model — in plain English:

A program that has read most of the internet and learned to predict "what text comes next" so well that it can write code, explain concepts, and solve problems.

It doesn't "think" like you do. It's incredibly good at pattern matching and generation.

No ML theory here. The key insight: LLMs are trained on massive text data and generate responses one token at a time. They're statistical but remarkably capable.

The AI Landscape

Copilots

Autocomplete on steroids.
Suggest code as you type.
Example: GitHub Copilot

Agents

Autonomous problem-solvers.
Read files, run commands, edit code.
Example: Claude Code

Key difference: Copilots assist while you drive. Agents can drive while you supervise.
Copilots = reactive, inline suggestions. Agents = proactive, can take multi-step actions. Both are useful, different situations.

Module 2

The Blank Slate

Understanding Context

This is the MOST important concept in the entire course. If they understand nothing else, they need to understand context.

Every conversation starts at zero

When you start a new AI conversation, the model knows nothing about your project, your code, your conventions, or your problem.

It's like hiring a brilliant contractor who shows up every morning with total amnesia.

This is the #1 mistake beginners make. They assume the AI "knows" their project. It doesn't. The contractor analogy really drives this home.

What is a Context Window?

The context window is the AI's short-term memory — everything it can "see" during a conversation.


ModelContext WindowRoughly...
Claude Sonnet 4.6200K tokens~500 pages of code
GPT-4o128K tokens~320 pages of code
Claude Haiku 4.5200K tokens~500 pages of code
Gemini 2.5 Pro1M tokens~2,500 pages of code

Bigger isn't always better — more context = slower + more expensive

What Goes Into the Context?

1. System prompt — Hidden instructions (tool rules, safety guidelines)

2. CLAUDE.md / config files — Your project instructions

3. Conversation history — Everything you've said + AI responses

4. Files read — Code the AI has opened and examined

5. Tool results — Output from searches, commands, tests

When the window fills up, oldest content gets compacted (summarized or dropped). Information can be lost!
Draw this as a bucket that fills up. When it overflows, stuff gets pushed out. This is why long conversations degrade.

Context: Claude Code vs Copilot

Claude Code

  • Reads CLAUDE.md at startup
  • Searches your codebase (grep, glob)
  • Opens files as needed
  • Sees git history
  • Runs commands, reads output
  • You control: what to share

GitHub Copilot

  • Sees the current file
  • Sees open tabs in your IDE
  • Reads neighboring files
  • Reads .github/copilot-instructions.md
  • Automatic: IDE decides what to send
  • Less control, but less effort
Key insight: Claude Code = explicit context (you guide it). Copilot = implicit context (IDE sends what it thinks is relevant). Both benefit from good project structure.

The Contractor Analogy

Before asking for help, give the AI what you'd give a new team member:

1. What project is this? — "A Flask REST API for a bookstore"

2. What's the tech stack? — "Python 3.11, SQLAlchemy, PostgreSQL"

3. What are the conventions? — "snake_case, ISO dates, wrapped responses"

4. What's the specific problem? — "GET /books returns 500 when DB is empty"

5. What does the error look like? — The actual traceback

Walk through each of these. Ask the audience: "What else would you tell a new contractor on day 1?" Write answers on a whiteboard.

Module 3

Your Project's README for AI

CLAUDE.md & Config Files

The Problem

You shouldn't have to re-explain your project conventions every single conversation.

Solution: Write it down once. The AI reads it automatically.

What is CLAUDE.md?

A markdown file that Claude Code reads at the start of every conversation.


Think of it as:

  • An onboarding doc for your AI assistant
  • A project README, but for the AI
  • Persistent memory across conversations

It goes into the context window automatically — no need to paste it every time.

CLAUDE.md Hierarchy

~/.claude/CLAUDE.md — Global (all projects)

Your personal preferences, coding style


./CLAUDE.md — Project root (this repo)

Project overview, tech stack, conventions


./src/CLAUDE.md — Directory-level (this folder)

Module-specific notes, patterns

All three are loaded. More specific files can add detail for subfolders.

Anatomy of a Good CLAUDE.md


# Bookstore API

## Overview
Flask REST API for managing a bookstore inventory.
Python 3.11, SQLAlchemy ORM, SQLite (dev) / PostgreSQL (prod).

## Conventions
- All endpoints use snake_case
- Dates are ISO 8601 format
- All responses: {"data": ..., "error": null}
- HTTP status codes: 200, 201, 400, 404, 500

## Commands
- Run: `flask run --debug`
- Test: `pytest -v`
- Lint: `ruff check .`

## Structure
- app.py - Routes and app factory
- models.py - SQLAlchemy models
- tests/ - Pytest test files
Walk through each section. Emphasize: concise, factual, actionable. Don't write an essay. Think "what would a new developer need?"

What to Include

Do Include

  • Project purpose (1-2 sentences)
  • Tech stack and versions
  • How to run, test, deploy
  • Naming conventions
  • Response/API formats
  • File structure overview
  • Non-obvious constraints

Don't Include

  • Entire API documentation
  • Full code examples
  • Meeting notes or history
  • Things derivable from code
  • Secrets or credentials
  • Overly long instructions
  • Contradictory rules
Remember: CLAUDE.md eats context window space. Every word counts. Be concise.

The Copilot Equivalent


# .github/copilot-instructions.md

You are assisting with a Python Flask REST API.
Use snake_case for all identifiers.
Always include type hints.
Use pytest for testing.
Wrap responses in {"data": ..., "error": ...} format.

GitHub Copilot reads this file for project-wide instructions.
Same idea, different file name.

Module 4

Talking to Machines

Prompt Engineering for Developers

Why Prompts Matter

The quality of AI output is directly proportional to the quality of your input.

A vague prompt gets a vague answer.
A specific prompt gets a specific, useful answer.

Anatomy of a Good Prompt

1. Task — What do you want? (be specific)

2. Context — What should the AI know? (tech stack, constraints)

3. Constraints — What should the AI NOT do? (boundaries)

4. Output format — How should the answer look? (code, explanation, list)


Task:       Add retry logic to the database connection in db.py
Context:    We use SQLAlchemy with PostgreSQL. Connection drops
            happen ~2x/day in production.
Constraints: Max 3 retries, exponential backoff, don't change
            the public API of get_connection().
Format:     Just the code changes, no explanation needed.

Pattern: Be Specific

Vague

Add error handling
Make this better
Fix the tests

Specific

Add try/except around the DB
call in get_user() that catches
ConnectionError and retries 3x
Refactor parse_csv() to use
pandas instead of manual string
splitting, keep the same return type
test_create_book fails with
"IntegrityError: NOT NULL" -
the test isn't setting the isbn field

Pattern: Step-by-Step Decomposition

For complex tasks, break them into steps:


I need to add user authentication to the Flask app. Let's do this
step by step:

1. First, add a User model to models.py with email, password_hash,
   and created_at fields
2. Add a POST /register endpoint that creates a user with a hashed
   password (use werkzeug.security)
3. Add a POST /login endpoint that returns a JWT token
4. Add a @require_auth decorator that validates the JWT
5. Add tests for all three endpoints

Start with step 1 and let me review before moving on.
This is great for complex tasks. You maintain control, can course-correct between steps, and get better results.

Pattern: Negative Constraints

Tell the AI what not to do:


Refactor the data processing pipeline in pipeline.py.

DO NOT:
- Change any function signatures (other code depends on them)
- Add new dependencies
- Modify the output format
- Touch the logging configuration

DO:
- Extract the repeated try/except blocks into a decorator
- Replace magic numbers with named constants
- Add type hints to all functions
Negative constraints prevent the AI from "helping" in ways you don't want.

Pattern: Few-Shot Examples

Show the AI what you want by example:


Convert these function docstrings to Google style. Here's the
pattern I want:

BEFORE:
  def add(a, b):
      """Adds two numbers and returns the result."""

AFTER:
  def add(a: int, b: int) -> int:
      """Add two numbers.

      Args:
          a: First number.
          b: Second number.

      Returns:
          The sum of a and b.
      """

Now apply this pattern to all functions in utils.py.

Reusable Prompt Templates

Save your best prompts as .md files:


prompts/
  code-review.md
  bug-investigation.md
  refactoring.md
  new-endpoint.md

Then paste or reference them when needed.
Build a team library of proven prompts.

In Claude Code, you can reference files directly:
> Review this code following the guidelines in prompts/code-review.md

Iterative Prompting

When the first answer isn't right, don't start over — refine:


Round 1: "Add input validation to the create_book endpoint"

AI adds basic validation but misses ISBN format


Round 2: "Good, but also validate that ISBN matches the ISBN-13 format (13 digits, starts with 978 or 979)"

AI adds ISBN validation


Round 3: "Now add a test for each validation rule"

Each round builds on the conversation context. The AI remembers what it did in previous rounds.

Common Anti-Patterns

Anti-PatternProblemFix
Too vague "Make it better" Define "better": faster? more readable? fewer bugs?
Too long 2-page prompt with every detail Focus on what's needed for THIS task
Contradictory "Keep it simple but handle every edge case" Prioritize: which matters more?
No acceptance criteria "Add a feature" (when is it done?) Define what success looks like

Choosing Your AI

Models, Cost & Performance

What Makes Models Different?

Three main factors:


Size (Parameters)
More parameters = more capable
but slower and more expensive


Training Data
What it learned from determines
what it's good at

Architecture & Fine-tuning
How it's built and optimized
for specific tasks


Reasoning capability
Some models "think" step-by-step
(o3, extended thinking)

The Claude Family

ModelBest ForSpeedCost
Haiku 4.5 Quick completions, simple tasks, high volume Very Fast $0.80 / $4 per 1M tokens
Sonnet 4.6 Balanced: code generation, review, explanation Fast $3 / $15 per 1M tokens
Opus 4.7 Complex reasoning, architecture, multi-file refactoring Slower $15 / $75 per 1M tokens

Costs shown as input / output per 1M tokens

Costs change - these are approximate. The key insight is the 10-20x cost difference between tiers. For most daily coding, Sonnet is the sweet spot.

OpenAI Models

ModelBest ForNotes
GPT-4o General coding, multimodal (images) Fast, good all-rounder
GPT-4.1 Coding tasks, instruction following Optimized for developer workflows
o3 Complex reasoning, math, hard bugs "Thinking" model - slower but more thorough
o4-mini Reasoning at lower cost Good balance of thinking + speed

Google & Open Source

ModelBest ForNotes
Gemini 2.5 Pro Large context tasks (1M tokens!) Great for analyzing entire codebases
Gemini 2.5 Flash Fast, cheap general tasks Good Haiku alternative
Llama 4 (Meta) Self-hosted, privacy-sensitive work Free, runs locally, good for sensitive code
DeepSeek V3 Cost-effective coding Strong coding performance, very cheap

Open source advantage: Run on your own hardware. Your code never leaves your network.

The Triangle: Pick Two

         QUALITY
          /\
         /  \
        /    \
       /______\
    SPEED    COST

Quality + Speed = Expensive
(Opus, GPT-4.1)

Quality + Cheap = Slow
(Reasoning models with wait)

Speed + Cheap = Lower quality
(Haiku, Flash, small models)

Decision Framework

TaskRecommended ModelWhy
Autocomplete / boilerplate Haiku / Copilot Speed matters most
Write a new function Sonnet / GPT-4o Good balance
Code review Sonnet / GPT-4.1 Needs understanding, not creativity
Complex refactoring Opus / GPT-4.1 Multi-file awareness needed
Debug a hard bug Opus / o3 Deep reasoning required
Explain code to me Sonnet / GPT-4o Good enough, save money
Emphasize: start with Sonnet/balanced model. Only go to Opus/o3 when the task actually requires deep reasoning. Most tasks don't.

What is a Token?

Tokens are how models measure text. Roughly:

  • 1 token ≈ 4 characters in English
  • 1 token ≈ 0.75 words
  • 100 lines of Python ≈ 500-800 tokens

Quick math: A 10,000-token conversation with Sonnet costs about $0.03 input + $0.15 output = $0.18

Costs add up with heavy use. But one hour of developer time costs far more than a day of AI tokens.

Switching Models in Practice

Claude Code:

> /model
# Select from available models
# Or: /model sonnet, /model opus, /model haiku

GitHub Copilot:

Settings > Copilot > Model selector in chat panel


Start every task with the cheapest model that could work. Upgrade only if the output quality isn't sufficient.

Plan Mode: Think Before Doing

> /plan

  login, and a decorator for protected endpoints.

In plan mode, Claude Code:

  1. Explores your codebase
  2. Designs an approach
  3. Presents a plan for your review
  4. Only implements after you approve
Use plan mode for any task that touches 3+ files or takes more than a few minutes.

How Claude Code Works

Under the hood, Claude Code follows a loop:


  1. Read — Examine relevant files (glob, grep, read)
  2. Think — Analyze the problem with full context
  3. Act — Edit files, run commands, create tests
  4. Verify — Run tests, check output
  5. Report — Tell you what it did

You can see each step happening in real time.
You approve or deny each action.

Permissions & Safety

Claude Code asks permission before:

  • Editing files
  • Running shell commands
  • Installing packages
  • Making network requests

Always review before approving. The AI can make mistakes. You are the final check.

Configure trust levels in .claude/settings.json to auto-approve safe operations.

When It Goes Off Track

SymptomFix
Wrong approach entirely /clear and start with a more specific prompt
Right idea, wrong details Correct it: "No, use X instead of Y"
Editing wrong files "Stop. Only modify app.py and tests/"
Over-engineering "Simpler. Just do X, nothing else."
Context seems confused /compact to summarize, or /clear
Think of it as pair programming. You're the senior dev — redirect early and often.

Copilot reads context from your IDE to generate suggestions:

Module 8

Wrap-Up

Key Takeaways & Continuous Learning

The 5 Things to Remember

  1. Context is everything — AI starts blank. Feed it well.
  2. Be specific — Vague prompts get vague answers.
  3. Use CLAUDE.md — Write it once, benefit every conversation.
  4. Right model, right task — Don't use Opus for autocomplete.
  5. You're still the developer — Review everything. AI assists, you decide.

Security Considerations

  • Never paste secrets into AI prompts (API keys, passwords, tokens)
  • Review generated code for security vulnerabilities (SQL injection, XSS)
  • Understand data flow — your code is sent to external APIs
  • Check licenses — AI-generated code may echo training data
  • Don't blindly run suggested shell commands

Treat AI-generated code the same way you'd treat a junior developer's PR: always review before merge.

Team Conventions

Things to agree on as a team:

  • Shared CLAUDE.md in every repo (commit it!)
  • A team prompt library (shared .md templates)
  • Model guidelines — which model for what
  • Review standards — AI code gets the same review as human code
  • Budget awareness — track token usage

Bonus

MCP

Connecting AI to Your Tools

What is MCP?

Model Context Protocol — A standard way to connect AI assistants to external tools and data sources.


Think of it as "plugins" for Claude Code:

  • Query a database directly
  • Read from internal APIs
  • Access documentation systems
  • Interact with cloud services

How MCP Works


    You (Claude Code)  <-->  MCP Server  <-->  External Tool
        "Query the         Translates        Database
         users table"      to SQL             (PostgreSQL)

MCP servers run locally and expose tools that Claude Code can call. You configure them in .claude/settings.json:


{
  "mcpServers": {
    "sqlite": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-sqlite", "books.db"]
    }
  }
}

When to Use MCP

Good Use Cases

  • Database exploration
  • Internal API integration
  • Documentation lookup
  • Cloud service management

Probably Not Worth It

  • Simple file-based projects
  • One-time tasks
  • When copy-paste is faster
  • Sensitive production systems
MCP shines when you repeatedly need AI to access the same external system.
